# seo-foundation Specification

## Purpose

Make the site discoverable by search engines and shareable on social platforms through valid sitemap, robots.txt, structured data, canonical URLs, and OG/Twitter meta tags.

## Requirements

### Requirement: XML Sitemap

The site MUST expose a valid `/sitemap.xml` generated by `@astrojs/sitemap` listing all public pages.

#### Scenario: Sitemap is accessible

- GIVEN the built site is deployed
- WHEN a crawler requests `/sitemap.xml`
- THEN it receives a 200 response with `Content-Type: application/xml`
- AND the XML validates against the sitemap 0.9 schema

#### Scenario: Sitemap references the canonical domain

- GIVEN `site` is set in `astro.config.mjs` to the production URL
- WHEN `/sitemap.xml` is generated
- THEN every `<loc>` entry uses the production domain (not `localhost`)

---

### Requirement: robots.txt

The site MUST serve a `/robots.txt` that allows all crawlers and references the sitemap URL.

#### Scenario: robots.txt is valid

- GIVEN the built site is deployed
- WHEN a crawler requests `/robots.txt`
- THEN it receives a 200 response containing `User-agent: *`, `Allow: /`, and a `Sitemap:` directive pointing to `/sitemap.xml`

---

### Requirement: JSON-LD Structured Data

Each page MUST include a `<script type="application/ld+json">` block with valid Schema.org markup.

The index page MUST include an `Organization` schema. Each product card SHOULD include a `Product` schema.

#### Scenario: Organization JSON-LD present on index

- GIVEN the index page is rendered
- WHEN its `<head>` is inspected
- THEN a `<script type="application/ld+json">` exists with `@type: "Organization"` containing name, URL, and logo

#### Scenario: JSON-LD passes Rich Results Test

- GIVEN the deployed index page
- WHEN validated with Google's Rich Results Test
- THEN no critical errors are reported

---

### Requirement: Canonical URL and OG/Twitter Meta Tags

`BaseLayout.astro` MUST output a `<link rel="canonical">` and Open Graph / Twitter Card `<meta>` tags using page-level props.

#### Scenario: Canonical tag uses production URL

- GIVEN a page is rendered with a `canonicalUrl` prop
- WHEN the `<head>` is inspected
- THEN `<link rel="canonical" href="{canonicalUrl}">` is present

#### Scenario: OG tags are populated

- GIVEN a page is rendered with `title`, `description`, and `ogImage` props
- WHEN the `<head>` is inspected
- THEN `og:title`, `og:description`, `og:image`, and `og:url` meta tags are present with the correct values

#### Scenario: Twitter Card tags are populated

- GIVEN a page is rendered with `title` and `ogImage` props
- WHEN the `<head>` is inspected
- THEN `twitter:card`, `twitter:title`, and `twitter:image` meta tags are present

#### Scenario: Missing ogImage uses default

- GIVEN a page is rendered without an `ogImage` prop
- WHEN the `<head>` is inspected
- THEN `og:image` falls back to `/og-image.png`
